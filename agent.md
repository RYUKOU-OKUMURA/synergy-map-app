# Agent Instructions

このプロジェクトでCodexまたはサブエージェントが作業するときの運用ルール。

## 基本方針

- 作業開始時は、まず`docs/project-context.md`を読む。
- すべてのドキュメントを毎回読まず、`docs/project-context.md`の読み分けルールに従って必要な関連資料だけを参照する。
- 要件定義は`docs/requirements.md`を参照する。
- 技術選定は`docs/tech-stack.md`を参照する。
- 実装計画は以下を参照する。
  - `docs/implementation-plan-phase-0.md`
  - `docs/implementation-plan-mvp-1.md`
  - `docs/implementation-plan-beta.md`
- MVP-1の範囲を超える実装は、明示的な指示がない限り行わない。
- MVP-1は「マップの材料入力 → AI抽出カード → 人間確認 → 顧客導線の売上マップ → 次に考える/動くためのMarkdown / CSV出力」を中心価値とする。
- MVP-1の主役ユーザーは、個人事業主・小規模会社の経営者。代表ユーザーは奥村さん自身とし、コンサルタント/支援者は副次ユーザーとして扱う。

## タスク進捗管理

- 実装計画ファイルのチェックボックスを進捗管理の正本にする。
- タスクが完了したら、該当する`- [ ]`を`- [x]`に更新する。
- 部分的に完了したタスクはチェックを入れず、必要なら短い補足を追加する。
- 新しいタスクが発生した場合は、該当フェーズの実装計画ファイルへチェックボックス付きで追加する。
- 仕様変更が発生した場合は、実装計画だけでなく`requirements.md`または`tech-stack.md`も更新する。

## コミット運用

- こまめにコミットする。
- 1コミットは1つの意味ある変更単位にする。
- 大きな変更を1コミットにまとめない。
- 実装、ドキュメント、整形だけの変更は可能な限り分ける。
- コミット前に、変更した実装計画ファイルのチェック状態を確認する。
- コミット前に、実行できる範囲のテスト・lint・型チェックを実行する。
- テストを実行できなかった場合は、コミット前に理由を記録する。

## サブエージェントレビュー

- コミット前には必ずサブエージェントでレビューを入れる。
- レビュー対象は、コミット予定の差分、関連する要件、関連する実装計画とする。
- サブエージェントには、以下を確認させる。
  - 要件から外れた実装になっていないか
  - MVP-1の範囲を超えていないか
  - セキュリティ・情報管理の方針に反していないか
  - AI出力schema、source chunks、item_sources、ai_runsの設計と矛盾していないか
  - テストまたは検証が不足していないか
- レビューで指摘が出た場合は、対応するか、対応しない理由を記録してからコミットする。
- レビュー未実施のままコミットしない。

## Phase 0運用

- Phase 0は技術ゲートであり、プロダクト機能を作り込むフェーズではない。
- Phase 0のGo条件を満たせない場合、MVP-1へ進む前に設計を見直す。
- Codex App Server、device-code認証、PDF / Excel読み取り、Typst日本語PDF、React Flow画像化、sidecar同梱は最優先で検証する。

## セキュリティ・情報管理

- 実クライアント資料をテストデータとして使わない。
- ログに資料本文、プロンプト全文、機密情報を残さない。
- AIへ送信する内容は、ユーザーが確認できる設計にする。
- MVP-1の既定は「要約だけ送る」とする。
- 案件削除時は、元資料、source chunks、ai_runs関連ファイル、exportsの扱いを確認する。

## 禁止事項

- MVP-1の範囲外機能を勝手に実装しない。
- ユーザーの既存変更を勝手に戻さない。
- `git reset --hard`や強制的なcheckoutを行わない。
- AI生成結果をschema検証なしにDB保存する設計にしない。
- frontendに広いfilesystem権限やshell権限を渡さない。
