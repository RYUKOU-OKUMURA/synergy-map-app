# MVP-1後の実装優先順位メモ

作成日: 2026-06-04

## このドキュメントの位置づけ

このドキュメントは、MVP-1完成後に何から着手するかを整理する優先順位メモである。

進捗チェックリストの正本は、引き続き `docs/plans/implementation-plan-mvp-1.md` と `docs/plans/implementation-plan-beta.md` とする。このファイルは、Beta計画の全項目を並べ替え、実装判断の順序を見失わないための補助資料として扱う。

## 基本方針

MVP-1直後は、機能を増やすより、まず「実事業で毎週使えるか」を確認する。

優先順位の判断軸:

1. 中心導線を壊さないか。
2. マップにたどり着くまでの距離を短くするか。
3. 根拠、推測、要確認を分ける力を強めるか。
4. 次に試す一手が具体化するか。
5. 継続利用の理由になるか。
6. スコープを増やさず検証できるか。

## P0: MVP-1完了判定と安定化

目的:

MVP-1を「一応動く」ではなく、代表ユーザーが実事業で一通り使える状態にする。

優先すること:

- 1週間の本人試験運用で、入力、抽出、マップ、次の一手、記録、出力が日常運用に耐えるか確認する。
- 実事業メモ、手入力URL、PDF、CSV、Excelで通し確認する。
- AI抽出カード、顧客導線マップ、AI視点、次に試す一手、確認事項、Markdown / CSV出力までを1セッションで確認する。
- 生成結果が `確定 / 推定 / 要確認` として読めるか確認する。
- Markdown / CSVが、次に考える記録として実際に使えるか確認する。
- source chunks、item_sources、ai_runsが検証に使えるか確認する。
- 削除、スナップショット、出力先フォールバックなど、ローカル保存周りの事故を確認する。
- macOS / Windowsの中心導線通し確認を行う。
- `pnpm build`、Rust test、schema validation、file parser testの実行環境を確認する。
- Rust testが環境要因で失敗する場合は、Beta着手前に原因と回避策を記録する。
- 抽出再実行やマップ再生成で既存カード、ノード、導線、レイアウト、AI視点が置き換わる範囲を、実事業データで確認する。
- 再生成前に名前付き保存できること、戻せること、削除対象がユーザーに分かることを確認する。

P0で増やさないこと:

- 新しいビュー。
- 新しいDBテーブル。
- 外部AI連携。
- PDF再開。
- URL / SNS自動読み取り。
- 選ばれる理由の仮説検査。

## P1: Beta前の基盤整理と小改善

目的:

MVP-1の中心価値を変えずに、Betaで機能を増やしても壊れにくい状態を作る。同時に、DBやUIを増やさず実装できる小改善だけを入れる。

優先候補:

1. 再生成時のデータ保全
   - 抽出再実行やマップ再生成で、何が消え、何が残るかを明示する。
   - 手修正済みカード、ノード位置、AI視点、確認事項を不用意に壊さない設計を検討する。
   - Betaで差分取り込みを行う前提を整理する。

2. ai_runs / Codex thread整合
   - AI実行履歴と、どのthread / 入力 / 出力が対応するかを実運用で追えるようにする。
   - 本格相談UIや外部AI調査メモ取り込みを入れる前に、ai_runsとthreadの扱いを揃える。

3. UI / command / backendの肥大対策
   - Tauri command呼び出しラッパー、状態管理、画面分割、Rust側DB処理の分離を必要な範囲で進める。
   - 大規模リファクタは避け、Beta追加で変更漏れが出やすい箇所から切り出す。
   - 既存migrationは編集せず、DB変更が必要な場合は追加migrationで対応する。
   - 2026-06-11実施:
     - [x] Vitest / jsdom とGitHub Actions CIを追加する。
     - [x] Tauri command呼び出しを `src/lib/api.ts` に集約する。
     - [x] Inspector編集フォームを `src/features/inspector/` に分割する。
     - [x] RustのMVP-1 wire DTOを `src-tauri/src/mvp1_types.rs` に分割する。
     - [x] Appシェル、Home、Projects、Settingsを `src/features/` 配下へ分割する。
     - [ ] App全体のView分割とprops削減は継続する。
     - [ ] Rust workspace loader / export / prompt処理の追加分割は継続する。

4. 追加調査用プロンプトのMarkdown補助セクション
   - 既存の確認事項、AI視点、次に試す一手から生成する。
   - DB、UI、Tauri commandは増やさない。
   - 詳細は `docs/future/hypothesis-validation-loop-future-concept.md` を参照する。

5. 試験運用で見えた入力/抽出/マップ表示の詰まり改善
   - 入力しづらい材料種別を整える。
   - 抽出カード修正の手間を減らす。
   - 大きいマップの見やすさを改善する。

6. 記録ビューと確認事項の運用改善
   - 未完了確認事項を見落とさない。
   - 施策と確認事項を、週次/月次の振り返りに使いやすくする。

P1の境界:

追加調査用プロンプトは、あくまでMarkdown出力の「考えるための記録」に留める。外部AIの回答を保存したり、マップへ反映したりしない。

## P2: Beta初期の実務化

目的:

MVP-1で見えた仮説を、継続運用に耐える作業導線へ変える。

優先候補:

1. プロンプトカード化
   - 確認項目ごとにコピー用プロンプトを出す。
   - 直接連携はしない。
   - ChatGPT / Claude / Perplexity別の細かいテンプレは、必要性が見えるまで増やさない。

2. 外部AI調査メモの手動保存
   - 調査結果を `source_files` / `source_chunks` の追加材料として保存する。
   - source種別は `external_ai_research_note` などにする。
   - 初期確度は必ず `推定` または `要確認` にする。

3. 会議中/思考中の軽い運用改善
   - 確認質問を追加、完了できるようにする。
   - 会議メモを残しやすくする。
   - 会議後にスナップショットへ残せるようにする。

4. バージョン保存/差分の初期判断
   - 実装する前に、週次/月次利用で差分比較が本当に必要か確認する。
   - 必要なら、名前付き保存と比較レポートの最小版から始める。

5. 配布準備の繰り上げ判断
   - 外部Betaをすぐ配る場合だけ、配布、署名、更新、sidecar更新方針をP2へ繰り上げる。
   - 代表ユーザー本人のローカル試験運用が中心なら、配布準備はP5に残す。

## P3: 根拠モデルと再分析ループ

目的:

外部調査や追加材料を、根拠チップと再分析へ安全に接続する。

優先候補:

- 外部AI調査結果を、事実候補、推測、要確認、根拠URLへ分類する。
- `source_claims` または `external_research_claims` のような中間モデルを検討する。
- URLは「根拠候補」として扱い、URL本文を確認するまで確定扱いしない。
- `item_sources` へ接続する前に人間確認を必須にする。
- 既存の抽出カードを全置換しない差分取り込みを設計する。
- 手修正済みカード、マップ、AI視点を壊さない再分析導線を作る。

P3で解決しない限り、外部AI調査結果による自動マップ更新は行わない。

## P4: 選ばれる理由の仮説検査

目的:

売上導線の手前にある「なぜあなたから買うのか」を、別ビューとして検査する。

初期スコープ:

- 主役は編集可能な一文。
- 根拠は既存の材料、抽出カード、発信、顧客の声、実績から拾う。
- 汎用性チェックは行うが、唯一性判定とは言わない。
- 出口は確認すべきことにする。
- 売上マップとの接続は、まず「眠っている売上資産」として扱う。

開始条件:

- MVP-1の売上マップが実事業で使われている。
- 追加調査用プロンプトまたは外部AI調査メモで、仮説を強める/弱める材料が扱える。
- 根拠と推測を分けるUIが十分に安定している。

P4で入れないこと:

- キャラ収入試算。
- 売上予測。
- ポジション偏差値。
- 唯一性スコア。
- 人格診断。
- 「あなたは唯一無二です」のような断定。

## P5: 大型拡張の判断

以下は、P0からP4の検証結果を見てから判断する。

- PDF出力強化
- OCR
- URL読み取り
- SNS/API連携
- 本格Codex相談UI
- クラウド保存/同期
- レイヤー型深掘りマップ
- 配布/自動更新

特にURL読み取りやSNS調査は、追加調査用プロンプトと混同しない。MVP-1とBeta初期では、URL本文を自動取得しない方針を維持する。

## 優先順位の要約

```text
P0 MVP-1通し確認と安定化
  → P1 データ保全 / ai_runs整合 / UI・backend整理 / 小さなMarkdown改善
  → P2 プロンプトカード / 外部AI調査メモ保存 / 会議中運用
  → P3 主張分解、根拠接続、差分再分析
  → P4 選ばれる理由の仮説検査
  → P5 PDF、OCR、URL、SNS、本格相談UI、クラウド、レイヤー型
```

## 保留してよいもの

以下は、MVP直後の最優先にはしない。

- PDF品質強化
- ダッシュボード型PDF
- 本格的な競合調査
- 自動URL/SNS読み取り
- 外部AI直接連携
- 自動マップ更新
- 大規模なレイヤー型マップ
- クラウド同期

理由は、入力、マップ生成、根拠確認、次の一手、記録出力という中心導線が安定する前に入れると、検証対象が増えすぎるためである。
