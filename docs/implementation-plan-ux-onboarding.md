# 初回マップ作成UX 実装計画

作成日: 2026-05-18

## 目的

`docs/ux-improvement-plan-initial-map-flow.md` の方針を、実装フェーズ、タスク、優先順位、完了条件に落とし込む。

この計画の最優先ゴールは、初回ユーザーが迷わず、5分以内に最初の売上マップへ到達できること。

この改善計画のゴールは、初回ユーザーが迷わず情報を入れ、Codex接続状態を理解したうえで、最初の売上マップを生成し、そのまま相談・施策検討に進める状態にすること。

最終的には、ユーザーが「何を入れればいいか分かる。押すボタンも分かる。マップが出る。そこから次に何を考えるべきかも分かる。」と感じる状態を目指す。

## 実装方針

大きくは2段階で進める。

- Phase 1: 迷わず最初のマップに到達する入口を作る。
- Phase 2: 情報ソース、Codex設定、生成後相談導線を強化する。

実装管理上は、Phase 0で土台整理、Phase 3で検証と磨き込みを行う。

## 現状の前提

- 初期表示は `src/App.tsx` の `view` 初期値が `map`。
- 空マップ時は `MapWorkspace` 内の `empty-map` が表示される。
- 新規マップ作成は `ProjectsView` の `新しいマップを作る` から行う。
- ファイル投入は `SourcesView` で PDF / CSV / XLSX / Markdown / Text のみ対応している。
- `run_extract_items` は読み取り済み `source_chunks` がないと失敗する。
- `generate_map_from_items` は採用中の抽出カードがないと失敗する。
- Codex接続確認用のTauri commandはあるが、主要UIには露出していない。
  - `get_codex_runtime_info`
  - `run_codex_smoke_test`
  - `run_codex_device_code_check`

## データ設計方針

非ファイル入力も、まずは既存の `source_files` / `source_chunks` パイプラインに乗せる。

理由:

- 既存のAI抽出、出典表示、AI履歴、マップ生成を再利用できる。
- URL / SNS / 自由メモ / 商品情報を「情報ソース」として同じ扱いにできる。
- 新しい分析パイプラインを増やさずに、初回UXを先に改善できる。

追加する入力種別:

- `onboarding_brief`
- `manual_note`
- `website_url`
- `sns_url`
- `product_info`

内部処理の考え方:

- ユーザー入力をローカルのMarkdownテキストとして保存する。
- `source_files.file_type` に入力種別を保存する。
- `source_files.metadata_json` に元URL、SNS種別、目的IDなどを保存する。
- `source_chunks` にCodexへ渡せる本文を保存する。
- URL / SNSはPhase 2初期ではクロールしない。ユーザーが入力したURLと補足メモを公開情報ソース候補として保存する。

必要であれば、目的や初回フロー状態を保持するために `project_onboarding` テーブルを追加する。

候補カラム:

- `project_id`
- `purpose_id`
- `purpose_label`
- `purpose_note`
- `hypothesis_mode`
- `information_level`
- `created_at`
- `updated_at`
