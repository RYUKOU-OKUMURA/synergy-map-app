# 仮説検査ループ将来構想

作成日: 2026-06-04

## このドキュメントの位置づけ

このドキュメントは、MVP-1後に検討する「追加調査用プロンプト」と「選ばれる理由の仮説検査」の構想メモである。

MVP-1の仕様書ではない。MVP-1の中心価値は、引き続き「マップの材料入力 → AI抽出カード → 人間確認 → 顧客導線の売上マップ → 次に考える/動くためのMarkdown / CSV出力」とする。

この構想は、MVP-1の売上導線ビューを拡張するための候補であり、既存の `docs/mvp-spec.md` や `docs/requirements.md` へ直接混ぜない。実装判断では、まず `docs/project-context.md` と該当フェーズの実装計画を優先する。

## MVP仕様本体に入れない理由

この構想は、MVP-1の中心導線を強化できる可能性がある一方で、扱いを誤ると別プロダクトに見える。

混ぜない理由:

- `docs/mvp-spec.md` に入れると、MVP-1で実装する機能と誤読される。
- `docs/requirements.md` に入れると、正式要件として確定したように見える。
- `CONTEXT.md` に用語として入れると、MVP-1の「顧客導線ビュー」の文脈が濁る。
- Beta計画の詳細チェックリストに入れすぎると、優先順位が未確定なのに実装約束に見える。
- 「選ばれる理由」や「ポジション」は、売上導線の前段にある別レイヤーであり、1枚目の売上マップへ直接混ぜるとビューの目的が曖昧になる。

したがって、このドキュメントは「MVP-1後に検討する構想メモ」として保存し、正式仕様化はMVP-1の実利用結果を見てから判断する。

## 中心コンセプト

シナジーマップは、売上導線上の仮説、詰まり、未接続の可能性、確認すべきことを出す。

この構想では、その仮説を外部AIや人間の追加調査に渡し、調査結果を再び「マップの材料」として戻す。

```text
マップの材料
  → 売上マップ
  → 仮説 / 確認すべきこと
  → 追加調査用プロンプト
  → 外部AIまたは人間による追加調査
  → 調査メモをマップの材料として再投入
  → 根拠 / 確度 / 要確認項目を更新
```

一文で表すなら、次の通り。

> シナジーマップが問いを作り、外部AIや人間が追加材料を集め、その材料をまたシナジーマップに戻す。

## 基本思想

外部AIは、答えを確定する場所ではない。

外部AIへ渡すものは「確認すべきことを調べるためのプロンプト」であり、戻ってくるものは「事実」ではなく「追加材料」である。

この扱いは、既存のプロダクト文法と同じである。

- 仮説であることを明示する。
- 根拠、推測、要確認を分ける。
- 売上改善を保証しない。
- 競合不在、市場で唯一、確実に選ばれる、といった断定をしない。
- 最終判断は、ユーザー確認とシナジーマップ側の再分析に戻す。

## 用語

MVP-1または初期Betaで使ってよい表現:

- 追加調査用プロンプト
- 確認事項を調べるためのプロンプト
- 追加材料を集めるためのプロンプト
- 外部AI調査メモ
- 選ばれる理由の仮説検査

避ける表現:

- 外部AIリサーチ
- AI競合調査
- 市場調査AI
- ポジション診断
- 売上改善調査
- 唯一性スコア
- ポジション偏差値
- キャラ収入診断

理由は、外部AIが主役に見えること、調査結果が正解に見えること、診断や保証のように見えることを避けるためである。

## 追加調査用プロンプト

### 目的

売上マップで出た仮説や確認事項を、外部AIまたは人間が追加調査しやすい形に変換する。

対象にする入力元:

- 確認質問
- 確認事項 / タスク
- 次に試す一手
- AI視点の「売上導線の欠陥」
- AI視点の「眠っている売上資産」
- AI視点の「利益化の盲点」

### MVP-1で許容する最小範囲

MVP-1に入れる場合でも、独立機能にはしない。Markdown出力の末尾に補助セクションとして追加するだけに留める。

許容範囲:

- Markdown出力に `## 追加調査用プロンプト` を追加する。
- 既存生成物だけを入力元にする。
- プロンプトは最大1から3本に制限する。
- 確認事項がない場合はセクションを出さない。
- 調査結果は事実ではなく追加材料として扱う、と明記する。
- 外部AIの回答を、再投入用の要点として整理する回答形式にする。

MVP-1では行わないこと:

- 新しい画面やタブを作る。
- 新しいDBテーブルを作る。
- Tauri commandを増やす。
- CSV出力に追加する。
- ChatGPT / Claude / Perplexity 別テンプレートを持つ。
- 外部AIへ直接連携する。
- 外部AI回答を貼り付けて取り込む。
- 外部AI回答でマップや確度を自動更新する。

### Markdown構成案

```md
## 追加調査用プロンプト

このセクションは、この売上マップで出た仮説や確認事項を、ChatGPT / Claude などで追加調査するためのプロンプトです。
外部AIの回答は事実として確定せず、追加のマップ材料として扱ってください。

### 調査したい仮説
- ...

### 貼り付け用プロンプト
以下の仮説について、追加材料を集める目的で調査してください。断定や売上保証は避け、事実・推測・追加確認事項を分けてください。

対象の仮説:
...

回答形式:
1. 確認できそうな事実
2. 根拠URL・参照すべき情報
3. 推測に留まること
4. 仮説を強める材料
5. 仮説を弱める材料
6. 人間が追加で確認すべきこと
7. シナジーマップに再投入するとよい要点
```

### プロンプトに必ず含める制約

- この仮説を事実として扱わない。
- 根拠がない場合は不明と書く。
- 一般論と、この事業に当てはまる可能性を分ける。
- 売上向上を保証しない。
- 競合不在、市場で唯一、確実に選ばれる等を断定しない。
- 回答は再投入用の追加材料として整理する。

## 選ばれる理由の仮説検査

### 位置づけ

これは「ポジション診断」ではない。

売上マップが見るものは、商品、集客、顧客接点、売上がどう流れているかである。一方で、選ばれる理由の仮説検査が見るものは、そもそもなぜユーザーから買う理由が生まれるのかである。

したがって、MVP-1の顧客導線ビューには混ぜない。Beta以降の別ビュー、または2枚目のマップ候補として扱う。

### 中心思想

> 席を授けない。席を疑う。

AIが「あなたの強みはこれです」と断定するのではなく、ユーザーと一緒に「この一文は本当に選ばれる理由になっているか」を検査する。

### v1で扱う要素

- ポジション仮説文
  - 主役は編集可能な一文。
  - AIは候補を出すだけで、選び直すのはユーザー。
- 根拠
  - 専門性、実績、発信、顧客の声、商品、過去の相談内容など。
  - 既存のマップの材料、抽出カード、source chunksと紐づける。
- 汎用性チェック
  - 同じ一文を他の同業者も簡単に名乗れるかを見る。
  - 「唯一性判定」と断定しない。
- 取得コスト
  - 年数、経験量、失敗、当事者性、専門性、顧客接点など。
- 確認すべきこと
  - 実際の顧客に聞く問い、プロフィールや商品説明で試す問いに落とす。

### 1枚目の売上マップとの接続

最初から線で接続しない。

初期接続は、AI視点の「眠っている売上資産」として扱う。

> 強い専門性や選ばれる理由の種があるが、まだ商品・発信・導線に接続されていない。

この扱いなら、顧客導線ビューを汚さず、2枚目の意味を出せる。

## Beta以降の段階案

### Beta 1: プロンプトカード

- 確認項目ごとのプロンプトカードを表示する。
- コピー操作だけを提供する。
- ChatGPT / Claude / Perplexity の直接連携はしない。
- 外部AI回答の貼り付け欄はまだ作らなくてよい。

### Beta 2: 外部AI調査メモの保存

- 調査結果をユーザーが手動で貼り付ける。
- `source_files` / `source_chunks` の追加材料として保存する。
- source種別は `external_ai_research_note` などにする。
- 日本語UIでは「外部AI調査メモ」と表現する。
- 初期ステータスは必ず `推定` または `要確認` にする。
- 自動で確定扱いしない。

### Beta 3: 主張分解と根拠接続

- 貼り付け結果を要約、分類する。
- 根拠URL、推測、追加確認事項を分離する。
- 必要に応じて `source_claims` または `external_research_claims` のような中間テーブルを追加する。
- 人間確認後に `extracted_items`、`item_sources`、根拠チップへ接続する。
- 再分析時の追加材料に含める。
- 手修正済みカードを壊さない差分取り込みを設計する。

### Beta 3後半以降: 選ばれる理由の仮説検査

- 一文、根拠、汎用性チェック、確認すべきことを別ビューとして試す。
- 追加調査ループの材料を使って、仮説を強める材料と弱める材料を分ける。
- 売上マップとの接続は、まず「眠っている売上資産」として表示する。
- キャラ収入試算、売上予測、唯一性スコアは入れない。

### Beta後半以降: 2枚目マップや導線接続

- 1枚目の売上マップに、選ばれる理由の仮説を線で直接接続するか検討する。
- 「未接続のポジション」を、どの商品、発信、顧客接点へ接続するか検討する。
- ただし、1枚目の顧客導線ビューが安定するまでは実装しない。
- 早期に線を出すと、売上導線ビューの読みやすさが落ちるため注意する。

## 技術メモ

MVP-1で追加調査用プロンプトを入れる場合は、Markdown生成処理に小さなセクション生成関数を足すだけでよい。

想定:

```rust
fn render_research_prompt_section(workspace: &ProjectWorkspace) -> Option<String>
```

`render_markdown` から呼び出す。`export_markdown`、出力ボタン、保存先、`export_jobs` は既存のまま使う。

Beta 2以降の外部AI調査メモは、既存の手入力情報ソースと同じく、Markdown化して `source_files` / `source_chunks` に保存できる。

ただし、Beta 3で主張単位に分解する場合、現状の `item_sources` だけでは以下の粒度が足りない可能性がある。

- 事実候補
- 推測
- 要確認
- 根拠URL
- URL本文を確認済みかどうか
- 人間確認済みかどうか

この段階では、中間モデルの追加と、既存抽出カードを全置換しない差分取り込みが必要になる。

## 禁止事項

MVP-1では禁止:

- 外部AI直接連携
- 自動ブラウジング
- URL読み取り
- SNS自動調査
- 競合自動調査
- 外部AI回答の貼り付け取り込み
- 外部AI回答による自動マップ更新
- 外部AI回答による確度の自動上昇
- 新規DBテーブル追加
- 新規画面 / タブ追加
- プロンプトテンプレ管理UI
- 収益試算
- 売上予測
- ポジション偏差値
- 唯一性スコア

Beta以降でも原則禁止または要注意:

- 根拠URLなしで確認済み扱いにする。
- 外部AI回答を事実としてラベル付けする。
- 競合不在、唯一、勝てる、売上が上がる等を断定する。
- ユーザーの専門性や人格を診断する表現を使う。
- 外部AIの回答をそのまま施策決定に使う導線を作る。

## 受け入れ条件の例

MVP-1に追加調査用プロンプトを入れる場合:

- Markdown出力に `追加調査用プロンプト` セクションが出る。
- 確認事項がない場合は出ない。
- セクション内に「外部AI回答は追加材料であり事実確定ではない」と明記される。
- 売上保証、競合不在、唯一性断定の文言が出ない。
- DBスキーマ、UI、Tauri commandを増やさない。
- rejectedの抽出カード、施策、AI視点を含めない。
- source chunk本文やローカルファイルパスをプロンプトに混ぜない。

## 次に判断すること

- MVP-1にMarkdown補助セクションとして入れるか、MVP後最初の小改善に回すか。
- プロンプトは1本にまとめるか、最大3本に分けるか。
- Beta 1のプロンプトカードを、出力画面に置くか、マップ内コンテキストパネルに置くか。
- 外部AI調査メモのsource種別名を `external_ai_research_note` にするか。
- 主張単位の中間テーブルを、Beta 3で本当に作るか。
