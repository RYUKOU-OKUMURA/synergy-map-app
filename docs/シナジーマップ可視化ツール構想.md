# シナジーマップ可視化ツール構想

作成日: 2026年5月15日

更新日: 2026年5月16日

## 概要

一つの企業が複数の事業・プロジェクト・集客チャネル・商品を持っているときに、それぞれがどうつながっているか、どこに未接続の可能性があるか、どの連携から着手すると相乗効果が出やすいかを可視化するツール。

単なる図解ツールではなく、経営者や右腕ポジションの人が、複数事業の関係性を整理し、次の打ち手を考えるための思考補助ツールとして設計する。

## 背景

東條さんとのAI顧問の会話で、複数事業を展開している企業ほど、事業間・媒体間・顧客接点間のつながりが見えにくくなっているという話が出た。

例として、ペット関連事業者では以下のように多くの事業やチャネルが存在していた。

- ペットショップ
- ECサイト
- トリミング
- ペットプール
- ペットホテル
- オンラインサロン
- YouTube
- Instagram
- TikTok
- ライブ配信
- 日々の売上スプレッドシート
- Google Analytics

それぞれが単体では動いているが、YouTubeがECや店舗来店にどうつながっているのか、SNSが既存顧客やオンラインサロンにどう波及しているのかが見えにくい。

この「点で動いている事業や施策」を、線や面として見えるようにするのがシナジーマップの狙い。

## 解決したい課題

- 複数事業がバラバラに動いていて、全体像が見えない。
- 集客チャネルと売上・利益の関係が見えない。
- YouTubeやSNSの活動が、どの事業に効いているのか説明しづらい。
- 既存の顧客接点を別事業へつなげられていない。
- 経営者の頭の中には関係性があるが、スタッフや外部支援者と共有しづらい。
- 改善候補が多すぎて、どこから手をつけるべきか判断しづらい。

## ツールの基本コンセプト

企業内の事業・プロジェクト・媒体・商品・顧客接点・データをノードとして登録し、それぞれの関係性をマップ化する。

そのうえで、AIが以下を分析する。

- すでに成果が出ている導線
- つながっているが弱い導線
- 本当はつなげられるのに未接続の導線
- 低コストで改善できそうな連携
- 売上・利益への波及が大きそうな連携
- 今はつなげなくてもよい関係

## 線の意味

ノード同士をつなぐ線には、関係性の種類や状態を持たせる。

- 太い線: すでに成果が出ている導線
- 細い線: 関係はあるが弱い導線
- 点線: 仮説段階の導線
- 赤い線: 詰まっている導線
- 緑の線: 改善余地が大きい導線
- 青い線: データ連携できている導線
- グレーの線: 現時点では優先度が低い導線

## AIに聞けること

このツールの価値は、マップを作るだけではなく、AIに関係性を読み解かせることにある。

質問例:

- この会社で一番シナジーが出そうな未接続ポイントを3つ出して。
- 今月やるなら、低コストで効果が出やすい連携はどれか。
- このプロジェクトを動かすと、どの事業に波及効果があるか。
- 逆に、今つなげても意味が薄いものはどれか。
- YouTubeの発信をEC売上につなげるには、どの導線を整えるべきか。
- 店舗利用者をオンラインサロンへつなげるなら、どの接点が自然か。
- 既存顧客に追加提案できる商品・サービスはどれか。
- 今あるデータだけで判断できることと、追加で取るべきデータは何か。

## MVP案

最初から立体的でかっこいいマップを作るより、まずはコンサル現場で実際に使える最小構成を作る。

MVPの中心価値は、外部サービス連携の多さではなく、以下の一連の流れが破綻せずに動くこと。

1. 資料を入れる。
2. AIが事業・商品・チャネル・顧客接点・財務参考情報を抽出する。
3. 抽出結果を人が確認する。
4. シナジーマップを生成する。
5. AIが現状分析、未接続シナジー、優先施策を出す。
6. コンサル側が資料として出力できる。

入口コピー:

> 3ステップで事業のつながりを見える化

サブコピー:

> 資料を入れるだけで、AIがシナジーマップと優先施策を整理します

### MVPの前提

- まずはコンサル側が使う専用ツールとして設計する。
- クライアント側のログイン、共同編集、閲覧リンクはMVPでは作らない。
- クライアント共有は、PDF、Markdown、CSVの出力で対応する。
- マインドマップや作図ツールの代替ではなく、AIが事業構造を解釈して改善仮説まで出す戦略補助ツールとして差別化する。
- 会議中は軽く動くことを優先し、重い資料読み取りや再分析は非同期で処理する。

### MVPで必ず検証すること

- 手動アップロードした資料をAIが読み取れるか。
- 読み取った内容が抽出カードとして整理されるか。
- 人が軽く修正した内容をマップに反映できるか。
- 顧客導線ビューと事業・収益ビューを切り替えられるか。
- Codexをアプリ内に組み込み、マップについて追加相談できるか。
- AIが施策カードとプロジェクト案を出せるか。
- レポート型PDF、ダッシュボード型PDF、Markdown、CSVとして出力できるか。
- バージョンを残し、過去版と比較できるか。

### Codex組み込み方針

アプリ内にはCodexを組み込み、ユーザーがシナジーマップについてそのまま相談できるようにする。

Codex App Serverは、Codexの会話、履歴、承認、イベントストリームを自社プロダクト内に深く組み込むための仕組みとして使う。

参考: https://developers.openai.com/codex/app-server

MVPでCodexに担わせたい役割:

- アップロード資料の読み取り補助
- 抽出カードの生成
- 未確認質問の生成
- シナジーマップの解釈
- 施策カードとプロジェクト案の生成
- マップ修正指示への対応
- レポート文面の生成

注意点:

- スプレッドシート、HP、SNS、会計データを読む機能そのものはアプリ側のツール・権限設計が必要。
- Codex App Serverだけで外部サービス連携が自動的に完結するわけではない。
- MVPでは、まず手動アップロード資料とアプリ内構造データをCodexに扱わせる。

### 3ステップ生成体験

#### Step 1: 資料を入れる

チェックリストとドラッグ&ドロップを併用する。

チェックリスト項目:

- 会社概要
- 事業・サービス一覧
- 商品一覧
- 主要顧客
- 集客チャネル
- HP/SNS URL
- 売上資料
- 決算書PDF
- 月次データ
- 過去施策・キャンペーン
- ヒアリングメモ

対応したい入力:

- PDF
- CSV
- Markdown
- テキスト
- 画像
- スプレッドシート書き出し
- 議事録
- 提案資料
- HP URL
- SNS URL

MVPでは外部サービスの自動連携よりも、手動アップロードを優先する。Google Analytics、YouTube Studio、SNSインサイト、freeeなどの認証連携はMVP後に回す。

#### Step 2: AI抽出結果を確認する

AIが読み取った内容を、いきなりマップ化せず、まず抽出カードとして確認する。

抽出カードの種類:

- 事業
- 商品・サービス
- 集客チャネル
- 顧客接点
- 財務参考情報
- データ資料

各カードで確認・修正できる項目:

- 名前
- 分類
- 短い説明
- 情報源
- 確定 / 推定 / 要確認
- 影響力 高 / 中 / 低
- 主観重要度 高 / 中 / 低
- 一言メモ

AI確認質問は、まずマップ生成に必須の質問から出す。次に財務判断に効く質問、最後にシナジー発見に効く質問を出す。

#### Step 3: シナジーマップを生成する

MVPでは以下の2つの表示モードを切り替えられるようにする。

- 顧客導線ビュー: 認知、興味、問い合わせ、来店、購入、継続、紹介の流れを見る。
- 事業・収益ビュー: 売上を生む事業・商品を中心に、集客チャネル、顧客接点、財務参考情報がどう接続しているかを見る。

会社・ブランド中心ビューはMVP後に追加する。

### 情報の種類

AIが扱う情報は、根拠の強さごとに分ける。

- 確定情報: コンサル側・クライアント側が明示入力した情報。
- 公開情報: HP、公開SNS、YouTubeチャンネルなどから取得した情報。
- 内部実績データ: 決算書、売上表、CSV、スプレッドシートなど、管理者側だけが持つ数字。
- 推定情報: AIが文脈から推測した関係性。
- 未確認仮説: 次回ヒアリングや提案で検証すべき可能性。

マップ上では、確定情報・公開情報・内部実績データ・推定情報・未確認仮説が区別できるようにする。特に、AIの推測と事実情報が混ざらないことを重視する。

### 情報充実度

「完成度」ではなく「情報充実度」として表示する。AIの正しさを保証する表現は避ける。

表示方法:

- カテゴリ別に表示する。
- 情報が薄いノードや線は、マップ上で薄色・点線・要確認バッジとして表示する。
- 追加すると分析精度が上がる不足資料リストを出す。

カテゴリ例:

- 事業情報
- 商品・サービス情報
- 集客情報
- 顧客情報
- 財務情報
- 実績データ

例:

- 情報充実度 35%: 事業構造は見えるが、売上・顧客導線の根拠が少ない。
- 情報充実度 60%: 主要な導線は分析可能。財務・実績データが不足。
- 情報充実度 85%: 事業、導線、財務、顧客接点までかなり分析可能。

### ノードの表示

ノードの大きさは「影響力」を表す。

MVPにおける影響力の定義:

1. 売上・利益貢献
2. 顧客接点の強さ
3. 現場の主観重要度
4. AIによる補助的な波及可能性

波及可能性だけでノードを大きくしすぎると現場感とズレるため、MVPでは売上・利益貢献と顧客接点の強さを優先する。

視覚表現:

- サイズ: 影響力
- 色の濃さ: 情報充実度
- 枠線: ノード種類
- バッジ: 売上貢献 / 顧客接点強 / 現場重要 / AI注目 / 財務あり / 要確認

### 線の表示

線は単なる接続ではなく、方向・強さ・状態・流れているものを持つ。

線の状態:

- 強い導線
- 弱い導線
- 仮説導線
- 詰まりあり
- 接続候補
- 未接続

線の方向:

- A → B
- B → A
- 双方向
- 方向未確認

MVPで線が表す流れ:

- 顧客の流れ
- 売上・購買の流れ
- 認知・信頼の流れ

視覚表現:

- 太さ: 流れの強さ
- 向き: 影響の方向
- 点線: 仮説・接続候補
- 赤系: 詰まり
- 薄い線: 根拠が薄い
- ラベル: 顧客 / 売上 / 認知

矢印のアニメーションや流れる速度表現は面白いが、MVPでは軽量性を優先し、将来の演出候補にする。

### 編集できる範囲

MVPでは作図ツールのような自由編集は作り込まない。軽い修正とAIへの指示で再構成できることを優先する。

手で修正できる範囲:

- ノード名
- ノード分類
- ノード説明
- 線の状態
- 優先度
- 主観重要度
- 一言メモ
- AI仮説への判定

AIへの指示例:

- この導線は弱い扱いにして。
- この商品を中心に再構成して。
- この仮説は保留にして。
- 顧客導線ビューで見直して。

### AIコメント

マップ生成直後のAIコメントは、以下の順番で出す。

1. 現状の全体像
2. 強い導線
3. 弱い・詰まっている導線
4. 未接続シナジー候補
5. 今すぐ取り組むべき施策
6. 追加で必要な情報

会議中は短いコメントと施策カードを中心にする。じっくり分析モードでは、コンサル提案文型の文章生成もできるようにする。

### 施策カード

MVPの施策カード項目:

- 施策名
- つなぐノード
- やること
- 期待インパクト
- 実装コスト
- 優先度
- 確認指標
- AIの根拠
- クライアントに確認すべき質問

優先度は数値スコアにしない。

表示:

- 経営者視点: 高 / 中 / 低
- コンサル視点: 高 / 中 / 低
- 根拠コメント

経営者視点では、コストの低さ、実行しやすさ、スタッフ負担の少なさ、短期回収の見込みを重視する。

コンサル視点では、売上インパクト、シナジーの大きさ、長期資産性、差別化、検証価値を重視する。

分析画面では、インパクト × コスト/実行しやすさ のマトリクスで施策を表示する。

### 財務情報の扱い

MVPでは財務情報を軽く扱う。

読み込み対象:

- 決算書PDF
- 財務CSV
- スプレッドシート
- 月次売上表
- 部門別・商品別の簡易データ

MVPでは、財務データを完全な会計分析に使うのではなく、シナジーマップ生成時の参考情報として扱う。

AIコメントで見たい観点:

- 売上構成上、どの事業が中心か。
- 粗利が高そうな領域はどこか。
- 固定費を増やす施策は慎重にすべきか。
- 低コストで既存資産を活かせる導線はどれか。
- 財務情報だけでは判断できない点は何か。

freee、マネーフォワード、弥生などのAPI/MCP連携はMVP後に回す。

### 会議中の使用感

MVPでは、会議中に完全なライブ生成を目指さない。

会議中にできること:

- 事前分析済みのマップを即表示する。
- ノード名、分類、線の状態、優先度を軽く修正する。
- 修正内容をもとにAIが短い再分析をする。
- 今の話を踏まえた優先施策を数十秒で更新する。

資料追加は可能にするが、読み取りとマップ反映は非同期でよい。

表示例:

- 読み取り中
- 分析待ち
- 次回更新に反映

### エクスポート

MVPではWord出力は入れない。

対応する出力:

- レポート型PDF
- ダッシュボード型PDF
- Markdown
- CSV

レポート型PDFの構成:

- 概要
- 使用資料
- 情報充実度
- シナジーマップ
- AI分析コメント
- 優先施策
- 追加確認事項
- 次回アクション

ダッシュボード型PDFの構成:

- シナジーマップ
- 情報充実度
- 優先施策一覧
- インパクト × 実行しやすさマトリクス
- AI注目ポイント
- バージョン情報

CSVは、ノード一覧、関係性一覧、施策候補の管理用にする。

Markdownは、Obsidian保存、AI再投入、手直し用にする。

### バージョン管理

バージョン管理は、単なる編集履歴ではなく、事業成長の履歴として扱う。

MVPで残すもの:

- AI生成・AI再生成ごとの自動保存
- 節目ごとの名前付き保存
- 初回分析版
- コンサル修正版
- クライアント確認後版
- 施策実行後版

各バージョンに残す内容:

- マップ
- ノード一覧
- 関係性一覧
- AI分析コメント
- 優先施策
- 判断根拠
- 参照した資料
- 財務参考情報
- コンサル側メモ
- クライアント確認事項

比較機能:

- 比較レポート型: 初回版と現在版を選び、AIが変化点を文章でまとめる。
- マップ差分型: 2つのマップを横並びで比較し、追加・削除・強化・弱体化を色や線で見る。

タイムライン型、成果証明型はMVP後に追加する。

### 保存設計

MVPではハイブリッド型を前提にする。

- アプリ内には、マップ生成に必要な軽い構造データを保存する。
- 元資料、PDF、CSV、大きいスプレッドシートはクラウド保存を想定する。
- AIが毎回すべての資料を読み直さず、抽出済み要約・構造化データを使う。
- 必要なときだけ元資料へ戻れるようにする。

データ階層:

- マップ用データ: ノード、線、分類、優先度、コメント。即時表示用。
- 分析用データ: 資料から抽出した要約、根拠、AI分析履歴。
- 元資料: PDF、画像、CSV、スプレッドシート、議事録、提案資料。

Google Drive、OneDrive、Dropboxなどのクラウド保存対応は重要。ただし、MVPではまず保存先の抽象化と外部エクスポートを優先する。


## 可視化イメージ

最初は2Dマップで十分。

将来的には、立体的なシナジーマップとして以下のような表現もできる。

- 事業レイヤー
- 集客レイヤー
- 顧客接点レイヤー
- 商品レイヤー
- データレイヤー
- 財務レイヤー

各レイヤーを縦方向に重ねることで、事業間のつながりだけでなく、集客から売上・利益までの流れを立体的に見られるようにする。

## 実務での使い方

### コンサル提案

顧客に対して、現状の事業や施策が点で動いていることを説明する。

例:

「御社は事業が多いですが、今はそれぞれが点で動いています」

「ここをつなぐと、YouTubeの発信がECと店舗来店に波及します」

「今月はこの2本の導線だけ整えましょう」

### 週次レビュー

週1回、シナジーマップを見ながら以下を確認する。

- 新しくつながった導線
- まだ弱い導線
- 今週改善する導線
- 保留にする導線
- 次回検証する指標

### AI顧問・右腕業務

経営者の頭の中にある関係性を可視化し、次にどこへ着手するかを一緒に決める。

AIは、マップをもとに選択肢を出すだけでなく、判断材料も出す。

## 既存のプロジェクト管理文脈との接続

このツールは、東條さんAI顧問で出てきた以下の文脈とつながる。

- プロジェクトごとの情報分離
- サブエージェントのチーム設計
- 判断軸の明文化
- 保留タスク管理
- 週次PDCA
- 財務、EC、SNS、YouTubeの横断分析

つまり、シナジーマップは単体ツールではなく、プロジェクト管理・AI導入・経営改善をつなぐ中心画面になり得る。

## 商品化の方向性

対象ユーザー:

- 複数事業を持つ中小企業
- 店舗とECを両方持つ事業者
- SNSやYouTubeをやっているが売上導線が整理できていない企業
- 右腕型コンサル
- 税理士、財務コンサル、事業伴走者
- AI導入支援者

提供価値:

- 複数事業の全体像を見える化できる。
- 未接続の売上導線を発見できる。
- 改善施策の優先順位を決めやすくなる。
- 顧客への提案資料として使える。
- AI顧問やコンサル支援の標準ツールにできる。

## MVP後の追加実装ロードマップ

MVP完成後は、機能を一気に広げるのではなく、コンサル現場での使用頻度と実装難易度を見ながら段階的に追加する。

### Phase 1: コンサル実務の継続運用を強化

優先度: 高

MVP直後に優先する。

- タイムライン型のバージョン表示
- 成果証明型レポート
- 月次・四半期レビュー版の保存
- 前回比較の自動コメント
- 施策実行後の変化記録
- レポートテンプレートの複数化

狙い:

単発診断ではなく、長期伴走で「関わったことで何が変わったか」を示せるようにする。

### Phase 2: クラウド保存・共有を強化

優先度: 高

- Google Drive連携
- OneDrive連携
- Dropbox連携
- クライアント別フォルダ管理
- 元資料と出力物の自動保存
- 共有用PDFの管理
- バックアップと復元

狙い:

データ消失を防ぎ、長期運用に耐える保存設計にする。アプリ自体は軽く保ち、重い元資料はクラウド側に置く。

### Phase 3: 内部データ・分析ツール連携

優先度: 中〜高

- Google Sheets連携
- Google Analytics連携
- YouTube Studioアナリティクス連携
- SNSインサイト連携
- EC売上データ連携
- 更新ボタンによる最新データ再取得

狙い:

手動アップロード中心から、半自動・自動更新型の戦略ダッシュボードへ進化させる。

### Phase 4: 財務レイヤーの強化

優先度: 中〜高

- 月次PL入力
- キャッシュフロー入力
- 手元資金、入金予定、支払予定、借入返済の管理
- 投資余力の判定
- 短期回収施策と中長期投資施策の切り分け
- freee、マネーフォワード、弥生などのCSV/API/MCP連携

狙い:

税理士、会計士、財務コンサルが、財務情報を自分の得意分野として活かしながら、経営戦略まで提案できるようにする。

### Phase 5: クライアント共有・共同利用

優先度: 中

- クライアント閲覧画面
- 共有リンク
- コメント機能
- 資料追加依頼
- 確認質問への回答フォーム
- 権限管理
- 共同編集

狙い:

コンサル側専用ツールから、クライアントとの共同整理ツールへ拡張する。ただし、MVPではここを作り込まない。

### Phase 6: 表示モードと演出の拡張

優先度: 中〜低

- 会社・ブランド中心ビュー
- 財務レイヤービュー
- データフロービュー
- 紹介・口コミビュー
- 3Dレイヤーマップ
- 矢印の流速アニメーション
- プレゼン用表示モード

狙い:

視覚的なわかりやすさとプレゼン効果を高める。ただし、MVPでは軽量性を優先する。

### Phase 7: アドバンス編集版

優先度: 低

- ノードの自由配置
- 線の自由編集
- レイアウトテンプレート
- 表示テーマ変更
- 高度なマップ編集

狙い:

自由に図を作りたいユーザー向け。ただし、マインドマップツールとの差別化を崩さないように注意する。

## 最初に作るなら

まずは、実際のWebアプリMVPに入る前に、要件とプロンプトの動作を小さく検証する。

初期ステップ:

1. サンプル企業を1社用意する。
2. ヒアリングメモ、会社資料、HP/SNS URL、決算PDFまたはCSVを用意する。
3. AI抽出用の項目定義を作る。
4. 抽出カードのサンプルを作る。
5. 顧客導線ビューと事業・収益ビューのサンプルマップを作る。
6. AIコメントと施策カードの出力例を作る。
7. レポート型PDF相当のMarkdownを作る。
8. ダッシュボード型PDF相当のMarkdownを作る。
9. そのうえで、Reactなどで3ステップ式の簡易Webアプリにする。

最初の実装で目指すのは、完成されたデザインではなく、資料投入からシナジーマップ生成までの思考フローが成立すること。

## 一言でまとめると

シナジーマップは、複数事業の関係性を見える化し、AIに相乗効果の候補と優先順位を出させるためのツール。

きれいな図を作ることが目的ではなく、経営者や右腕が「どことどこをつなげると事業全体が伸びるのか」を考えるための実務ツールとして作る。
