# 初回マップ作成UX改善計画

作成日: 2026-05-18

## 目的

初回ユーザーが迷わず、5分以内に最初のシナジーマップへ到達できる状態を作る。

この改善計画のゴールは、初回ユーザーが迷わず情報を入れ、Codex接続状態を理解したうえで、最初のシナジーマップを生成し、そのまま相談・施策検討に進める状態にすること。

最終的には、ユーザーが「何を入れればいいか分かる。押すボタンも分かる。マップが出る。そこから次に何を考えるべきかも分かる。」と感じる状態を目指す。

現状は、初期表示が空のマップ画面である一方、実際には「案件作成 → 情報投入 → AI抽出 → マップ生成」という前段が必要になる。そのため、開発者でも「何をどこに入力すればマップを生成できるのか」が分かりにくい。

この改善では、空のマップ画面を単なる未生成表示ではなく、「新しいマップを作る」ための入口に変える。

## 基本方針

- アプリの入口は「案件管理」ではなく「マップ作成」に寄せる。
- ユーザーに「資料を投入してください」ではなく「マップの材料を追加してください」と伝える。
- 目的選択をCodexへの分析指示として扱う。
- 情報ソースはCodexが分析する根拠として扱う。
- 情報が少ない状態でも仮説マップ生成は許可する。
- 仮説が多い場合は、情報量と確度を明示する。

## 初回フロー

1. 目的を選ぶ。
2. 基本情報を入力する。
3. Codex接続状態を確認する。
4. マップの材料を追加する。
5. AIがマップの材料を整理する。
6. シナジーマップを生成する。
7. マップを見ながらCodexに相談する。

## 目的選択

ユーザーが迷わないよう、目的は自由入力だけでなく選択式にする。

- 売上導線を整理したい
- SNS / Webから売上につなげたい
- 複数事業のシナジーを見たい
- 既存顧客への追加提案を考えたい
- 事業の全体像を整理したい
- 提案前のヒアリング項目を出したい

目的は表示上の分類ではなく、Codexへの分析観点として扱う。同じ情報ソースでも、目的によって強調するノード、導線、確認質問、施策候補が変わる状態を目指す。

## 基本情報

最初の入力は軽くする。

必須:

- 企業名 / 案件名
- マップ生成の目的

推奨:

- 業種
- 今わかっている事業情報
- 今困っていること、見たいこと
- 自由メモ

## 情報ソース追加

「資料投入」ではなく「情報ソース追加」として扱う。

初期対応:

- ファイル
- 自由メモ
- ホームページURL
- SNSアカウントURL
- 商品 / サービス情報

URLやSNSの自動取得は後続実装でもよい。まずは、ユーザーが入力できる場所を用意し、Codexへの文脈として扱える状態を優先する。

## Codex接続確認

ユーザーが自分のChatGPTアカウントやCodexを使って分析できることが、このツールの価値に直結する。そのため、Codex接続は隠れた前提にせず、初回マップ作成フロー内で状態を明示する。

初回フロー内では、マップ生成前に接続チェックカードを表示する。

表示する状態:

- Codex CLI: 検出済み / 未検出
- Codex CLIバージョン
- ChatGPT認証: 接続済み / 未接続
- 接続テスト: 成功 / 失敗 / 未実行
- AI実行モード: Codex生成 / ローカルドラフト

主要操作:

- ChatGPTアカウントで接続
- 接続を確認
- Codex CLIの状態を再確認
- 接続できない場合の対処を見る

接続できていない場合でも、アプリ自体を止めない。AI生成や相談機能は制限し、ローカルドラフトにフォールバックする場合は明示する。

表示例:

- Codexに接続されています。ChatGPTアカウントを使ってマップを生成できます。
- Codexに未接続です。このまま進むとローカルドラフトで仮説マップを作成します。
- Codex CLIが見つかりません。インストール後に再確認してください。

また、初回フローとは別に、左ナビまたは案件外の共通領域に `設定` 画面を追加し、いつでもCodex接続状態を確認・再接続できるようにする。

設定画面に置く項目:

- Codex接続状態
- Codex CLIのパス
- Codex CLIバージョン
- ChatGPTログイン状態
- 接続テスト結果
- ローカルドラフトへのフォールバック有無
- エラー時の対処案

## 生成モード

### 根拠ありマップ

ファイル、URL、SNS、商品情報、自由メモなど、何らかの情報ソースがある状態。通常はこちらを推奨する。

表示例:

- 情報量: 中 / 高
- 確度: 根拠あり
- ボタン: シナジーマップを生成する

### 仮説マップ

企業名、目的、少量のメモだけで生成する状態。入口は軽くするが、推定であることを明示する。

表示例:

- 情報量: 低
- 確度: 仮説多め
- 警告: このマップは入力情報が少ないため、推測を含みます。資料やURLを追加すると精度が上がります。
- ボタン: 仮説マップを生成する
- 生成後CTA: 根拠を追加して精度を上げる

## CTA設計

`AIで更新` は抽象度が高いため、状態に応じて次の行動を直接示す。

- 情報がない: 情報を追加する
- source chunksがある: AIでマップの材料を整理する
- 抽出カードがある: シナジーマップを生成する
- マップがある: 施策と確認質問を生成する
- マップ生成後: このマップについて相談する

## 新規案件作成の扱い

ユーザー向けの言葉は `新規案件` ではなく `新しいマップを作る` に寄せる。

内部的には案件作成でよいが、ユーザー体験としては「目的選択 → 基本情報入力 → Codex接続確認 → 情報ソース追加 → AI整理 → マップ生成」の流れに統合する。

既存の案件一覧は残す。ただし初回体験の主役ではなく、過去のマップを開く場所として扱う。

## マップ生成後の体験

マップ生成で終わらせず、次の行動に接続する。

- Codexに聞く
- 強い導線を見る
- 詰まりを見る
- 未接続シナジーを見る
- 売上への一手を出す
- 次に確認すべき質問を出す

月額課金の継続価値は、マップを出すことだけではなく、マップを使って現状理解と次の施策を深掘れることにある。

## 実装優先順位

1. 空マップ画面をマップ作成フローに差し替える。
2. 目的選択UIを追加する。
3. Codex接続チェックカードを初回フローに追加する。
4. Codex接続設定画面を追加する。
5. 企業名、目的、メモだけで仮説生成できる入口を作る。
6. 情報ソース追加UIをファイル、メモ、URL、SNS、商品情報に広げる。
7. 情報量メーターと仮説警告を表示する。
8. CTAを状態別ラベルに変える。
9. 目的をCodexプロンプトに反映する。
10. マップ生成後の相談導線を強化する。

## 完了条件

- 新規ユーザーが最初に何を入力すればよいか分かる。
- 企業名、目的、メモだけでも仮説マップ生成へ進める。
- 情報ソースを追加すると精度が上がることが画面上で伝わる。
- Codex接続状態、ChatGPT認証状態、ローカルドラフトへのフォールバック有無が分かる。
- Codex未接続時に、ユーザーがどこで接続・再確認すればよいか分かる。
- `AIで更新` のような抽象的CTAに頼らず、次にやることがボタン名で分かる。
- 目的に応じてCodexの分析観点が変わる。
- マップ生成後にCodex相談、施策、確認質問へ自然に進める。
